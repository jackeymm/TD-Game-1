# 产品需求文档（PRD）

## 项目名称

Tower Defense Mini Game（塔防小游戏 MVP）

---

## 一、项目目标

开发一个**最小可玩的塔防小游戏（MVP）**，验证以下核心能力：

1. 游戏基础循环是否成立（建塔 → 防御 → 击败敌人）
2. 用户操作是否直观（点击建塔）
3. 游戏是否具备基本可玩性（胜负机制）

---

## 二、产品定位

* 类型：单机塔防小游戏
* 平台：微信小程序（或小游戏）
* 网络：纯本地运行（不依赖后端）
* 用户：单人体验

---

## 三、MVP范围（必须实现）

### 1. 地图系统

* 固定单地图
* 单条预设路径（数组路径点）
* 敌人严格按路径移动
* 不实现寻路算法

---

### 2. 敌人系统

#### 类型

| 类型   | 血量  | 速度  |
| ---- | --- | --- |
| 普通敌人 | 100 | 1.0 |

#### 属性

```js
{
  x: number,
  y: number,
  hp: number,
  speed: number,
  pathIndex: number
}
```

#### 行为

* 沿路径移动
* 到终点扣 1 点生命
* 被攻击减少 hp
* hp ≤ 0 时死亡并掉落金币（+10）

---

### 3. 防御塔系统

#### 塔类型

| 类型  | 攻击 | 范围  | 攻速     | 价格 |
| --- | -- | --- | ------ | -- |
| 普通塔 | 20 | 100 | 1000ms | 50 |
| 快速塔 | 10 | 80  | 500ms  | 50 |

#### 数据结构

```js
{
  x: number,
  y: number,
  range: number,
  attack: number,
  attackInterval: number,
  lastAttackTime: number
}
```

#### 行为

* 自动寻找范围内最近敌人
* 按 attackInterval 攻击
* 不需要玩家操作

---

### 4. 攻击系统

* MVP采用：**直接扣血模型**
* 不实现复杂子弹碰撞
* 后续可扩展子弹系统

---

### 5. 资源系统

#### 初始值

* 金币：100
* 生命：10

#### 规则

* 击杀敌人：+10金币
* 建塔：消耗金币
* 敌人到终点：-1生命

---

### 6. 波次系统

* 共 3 波
* 每波敌人数量递增：

| 波次 | 数量 |
| -- | -- |
| 1  | 5  |
| 2  | 8  |
| 3  | 12 |

* 每波结束自动进入下一波

---

### 7. 胜负系统

#### 胜利

* 所有波次完成

#### 失败

* 生命 ≤ 0

---

### 8. 操作系统

* 点击空地建塔
* 点击按钮选择塔类型
* 不支持拖拽

---

### 9. UI结构

#### 顶部 HUD

* 生命值
* 金币
* 当前波次

#### 中部

* Canvas 游戏画布

#### 底部

* 建塔按钮（2种塔）

---

### 10. 游戏循环（核心）

```text
初始化游戏
→ 生成当前波敌人
→ 游戏循环（每帧执行）：
    - 更新敌人位置
    - 塔检测并攻击敌人
    - 处理敌人死亡
    - 更新金币/生命
    - 判断波次是否结束
→ 是否还有下一波
    → 是：进入下一波
    → 否：胜利
→ 生命 ≤ 0：失败
```

---

## 四、非目标（明确不做）

* 多地图
* 升级系统
* 技能系统
* 商店
* 存档
* 排行榜
* 联网
* 音效
* 粒子特效
* 拖拽建塔

---

## 五、技术约束

* 使用微信小程序 Canvas
* 所有逻辑在前端实现
* 不使用第三方游戏引擎

---

## 六、全局状态结构

```js
{
  gold: number,
  life: number,
  wave: number,
  enemies: Enemy[],
  towers: Tower[]
}
```

---

## 七、验收标准

### 功能

* [ ] 页面能打开
* [ ] 敌人能移动
* [ ] 敌人到终点扣血
* [ ] 可以建塔
* [ ] 塔自动攻击
* [ ] 敌人死亡掉金币
* [ ] 波次推进正常
* [ ] 游戏可胜利/失败

### 可玩性

* [ ] 玩家能理解玩法
* [ ] 能完整打完一局
* [ ] 无报错

---

## 八、开发阶段

### Phase 1

* 页面 + Canvas
* HUD

### Phase 2

* 敌人路径移动

### Phase 3

* 建塔 + 攻击

### Phase 4

* 波次 + 胜负

### Phase 5

* 简单优化

---

## 九、开发约束（重要）

1. 每次只实现一个功能
2. 禁止一次性重构全部代码
3. 必须限定修改文件范围
4. 每一步必须可运行
5. 优先“能跑”，再优化

---

## 十、版本目标

👉 做出“可玩的塔防游戏”，不是完整游戏

判断标准：

* 能建塔
* 能打怪
* 有胜负
* 能玩完一局
